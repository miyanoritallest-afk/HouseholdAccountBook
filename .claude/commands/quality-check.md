# 品質チェック (Quality Check)

バックエンド（Rails API）とフロントエンド（Next.js）の実装を、以下のチェックリストに沿って品質レビューしてください。

## 使い方

```
/quality-check
```

または特定の対象を指定：

```
/quality-check backend
/quality-check frontend
/quality-check security
```

---

## チェックリスト

### 🔴 セキュリティ（必ず確認）

#### バックエンド
- [ ] **BE-SEC-1** JWTシークレットにフォールバック値がないこと（`ENV.fetch("KEY")` でデフォルトなし）
- [ ] **BE-SEC-2** 全APIエンドポイントに `authenticate_user!` が適用されていること
- [ ] **BE-SEC-3** リソースへのアクセスが必ずユーザースコープ経由であること（`Category.find` ではなく `current_user.categories.find`）
- [ ] **BE-SEC-4** Strong Parameters が `params.require(:key).permit(...)` 形式で使われていること（二重対応の排除）
- [ ] **BE-SEC-5** 日付パラメータなど外部入力にバリデーションがあること（不正値で500が返らないこと）
- [ ] **BE-SEC-6** パスワードがハッシュ化されていること（bcrypt 等）

#### フロントエンド
- [ ] **FE-SEC-1** `eval` / `new Function()` を使っていないこと
- [ ] **FE-SEC-2** dangerouslySetInnerHTML を使っていないこと（使う場合はサニタイズ必須）
- [ ] **FE-SEC-3** JWTトークンの Cookie に `SameSite=Strict` が設定されていること
- [ ] **FE-SEC-4** APIキーや秘密情報がクライアントサイドコードに含まれていないこと（`NEXT_PUBLIC_` 変数に秘密情報を入れない）

---

### 🟡 堅牢性・エラーハンドリング

#### バックエンド
- [ ] **BE-ROB-1** `rescue_from` で `RecordNotFound`, `ParameterMissing`, `ArgumentError` をキャッチしていること
- [ ] **BE-ROB-2** APIエラーレスポンス形式が統一されていること（`{ errors: [...] }` 配列形式）
- [ ] **BE-ROB-3** N+1クエリがないこと（ループ内で `find` を呼んでいないこと）
- [ ] **BE-ROB-4** `includes(:association)` で必要な関連をプリロードしていること

#### フロントエンド
- [ ] **FE-ROB-1** axios / fetch に `timeout` が設定されていること（推奨: 10000ms）
- [ ] **FE-ROB-2** 全APIコールに `try/catch` があること
- [ ] **FE-ROB-3** エラーメッセージをユーザーに表示していること（サイレント失敗がないこと）
- [ ] **FE-ROB-4** ローディング状態を管理していること（二重送信防止）

---

### 🟡 設計・コード品質

#### バックエンド
- [ ] **BE-DES-1** コントローラーがRESTfulなアクションのみ持っていること
- [ ] **BE-DES-2** ビジネスロジックがモデルまたはサービスオブジェクトに集約されていること
- [ ] **BE-DES-3** モデルにバリデーションが定義されていること
- [ ] **BE-DES-4** マイグレーションに外部キー制約があること

#### フロントエンド
- [ ] **FE-DES-1** APIコールがカスタムフック（`useXxx`）またはサービス層に集約されていること
- [ ] **FE-DES-2** コンポーネントが Props を通じてのみデータを受け取っていること（グローバル状態の濫用がないこと）
- [ ] **FE-DES-3** TypeScript の `any` 型を使っていないこと
- [ ] **FE-DES-4** フォームバリデーションがクライアント側にあること

---

### 🟡 アクセシビリティ（A11y）

- [ ] **A11Y-1** モーダルに `role="dialog"` と `aria-modal="true"` があること
- [ ] **A11Y-2** モーダルにフォーカストラップが実装されていること（Tab/Shift+Tab がモーダル内に閉じること）
- [ ] **A11Y-3** モーダルを Escape キーで閉じられること
- [ ] **A11Y-4** ボタンに意味のあるラベルがあること（アイコンのみのボタンには `aria-label`）
- [ ] **A11Y-5** フォームフィールドに `label` または `aria-label` が紐付いていること

---

### 🟢 テスト

- [ ] **TEST-1** バックエンドのモデルspecが実装されていること（pendingのみでないこと）
- [ ] **TEST-2** バックエンドのリクエストspecで認証・認可を検証していること
- [ ] **TEST-3** フロントエンドのE2Eテストが主要フローをカバーしていること

---

### 🟢 環境・設定

- [ ] **ENV-1** 秘密情報が環境変数で管理されていること（`.env` ファイルがgit管理外であること）
- [ ] **ENV-2** デフォルト秘密鍵（`changeme_*` 等）がコードに残っていないこと
- [ ] **ENV-3** 本番用の CORS 許可オリジンが明示的に設定されていること
- [ ] **ENV-4** 本番環境で詳細なエラーメッセージが露出しないこと（`Rails.env.production?` での分岐）

---

## レビュー手順

1. 対象プロジェクトのバックエンド・フロントエンドの主要ファイルを読む
2. 上記チェックリストの各項目を確認する
3. 問題が見つかった場合は以下の形式で報告する：
   - **ID**: チェックID（例: BE-SEC-3）
   - **ファイル**: 問題箇所のファイルパスと行番号
   - **問題**: 何が問題か
   - **修正案**: どう直すべきか
4. 問題がなければ ✅ として報告する
